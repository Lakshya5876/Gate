# 04 ÔÇö Runbook

_What this covers: the operational procedures ÔÇö install (greenfield/brownfield), `/init-governance`, `--upgrade`, uninstall, the release process, branch strategy, and how to verify a repo is actually governed._

All commands assume you cloned the framework once to a persistent path (e.g. `~/tools/Gate`) and run it *from inside your target repo*.

## Install

The installer always writes into **the repo you're standing in** (resolved via `git rev-parse --show-toplevel`, `install.sh:777`). It **refuses** to run against the `Gate` repo itself (`install.sh:780-784`).

### Preflight (both baskets)
- `git` and `python3` are hard requirements; the installer fails fast if either is missing (`install.sh:769-770`). The Claude Code CLI is only *warned* about (`install.sh:791-794`) ÔÇö you won't discover it's missing until `/init-governance`. Install it up front.
- A best-effort "your clone is stale" check runs unless `AI_DEV_WORKFLOW_SKIP_STALENESS_CHECK=1` (`install.sh:478-511`).

### Greenfield
```bash
mkdir my-new-project && cd my-new-project
git init && git checkout -b chore/claude-init
git commit --allow-empty -m "chore: repository birth"
~/tools/Gate/install.sh          # choose [g] when prompted
```

### Brownfield
```bash
cd /path/to/existing-repo
# Verify the 1M-LOC ceiling FIRST (basket README has the exact find command):
find . -type f -not -path "*/.git/*" -not -path "*/node_modules/*" \
  -not -path "*/.venv/*" -not -path "*/__pycache__/*" -not -path "*/vendor/*" \
  -not -path "*/dist/*" -not -path "*/build/*" -not -path "*/.next/*" \
  | xargs wc -l 2>/dev/null | tail -1
git checkout -b chore/claude-init
~/tools/Gate/install.sh          # choose [b] when prompted
```
The installer also does its own LOC check for brownfield and hard-errors above 1,000,000 (`install.sh:812-820`).

### The interactive `/dev/tty` basket prompt
Basket selection reads from `/dev/tty` directly, not stdin (`install.sh:802`): `read -r -p "Enter g or b: " BASKET_INPUT </dev/tty`. This means you **must run the installer from a real interactive terminal** ÔÇö piping input (`echo g | ./install.sh`) will not answer the prompt, and a CI/non-TTY context will fail here. Same pattern is used by `_confirm` (`install.sh:66-73`) for the upgrade/deprecation prompts, and by the SKIP_GATE bypass prompt. (In automated tests this is driven through a real pty; see `tests/gate/test_helper.bash`.)

### What a fresh install produces (target repo)
`.githooks/` (gate.sh, pre-commit, pre-push, verify_governance_integrity.sh) + `git config core.hooksPath .githooks`; `.claude/` (gate_state.json, settings.json with deny-list + hooks, hooks/, checkpoint_tool.py, checkpoints/, commands/init-governance.md + checkpoint-search.md, gate_integrity.sha256, session_state.json, and ÔÇö brownfield ÔÇö an unpopulated baseline.json); `.github/workflows/gate.yml`; `.github/CODEOWNERS` (placeholder team); the dev-guide + init-package `.md` at repo root; `.mcp.json` (if the graph server built); bypass-note refspecs in git config; `~/.claude/org_policy.json` (global). Full summary is printed by the installer (`install.sh:1144-1216`).

## `/init-governance` (the one manual step)

After install, open Claude Code in the target repo and run:
```
/init-governance
```
This slash command was generated by `install.sh:_write_init_command` (`install.sh:334-362`) from the `PROMPT STARTÔÇĞPROMPT END` block of the basket's implementation package ÔÇö so there is no file to copy-paste. Claude Code then interrogates you (product/stack/risk/debt), drafts spec docs under `docs/` for **your explicit approval** (greenfield), generates the repo-specific `CLAUDE.md`, fills stack-specific gate commands and (brownfield) the debt baseline, and completes the scaffold. Commit the result through a PR.

## Upgrade

From inside a governed repo (requires an existing `.claude/gate_state.json`, `install.sh:600`):
```bash
~/tools/Gate/install.sh --upgrade
```
It re-copies hooks, backfills trust-root settings for older installs, refreshes the dev-guide/init-package, re-pins the integrity manifest, **force-overwrites** `.github/workflows/gate.yml` (unlike fresh install, which skips if present), and bumps `framework_version`/`framework_last_upgrade` in `gate_state.json`. It **preserves** `baseline.json`, `CLAUDE.md`, `.mcp.json`, and `gate_state.json`'s receipts/token/thresholds/core_files. It never silently edits `CLAUDE.md`: if the dev guide's *content* changed it generates `/reconcile-governance` (propose-and-approve), and it offers (never forces) deprecation cleanup. Full matrix in `docs/UPGRADE.md`; code at `install.sh:597-740`.

**Critical:** re-pinning the manifest means **CI will fail until the new `.claude/gate_integrity.sha256` is committed alongside the files it covers** (`install.sh:732-738`). Use the exact commit command the upgrade prints ÔÇö it lists only what that run touched.

Verify:
```bash
python3 -c "import json; print(json.load(open('.claude/gate_state.json'))['framework_version'])"
```

## Uninstall

From inside the governed repo:
```bash
~/tools/Gate/uninstall.sh
```
It detects a governed repo via **any** known artifact (not just `gate_state.json`, which is often gitignored) (`uninstall.sh:169-179`), prints a full removal manifest, takes one blanket confirmation, then removes local artifacts, git config (`core.hooksPath`, bypass refspecs), framework `.gitignore` entries, and global resources (`~/.claude/org_policy.json` only if unmodified; the `code-review-graph` pipx package). It **never** auto-removes human-authored content: `CLAUDE.md` and `docs/` spec files get their own separate prompts (`uninstall.sh:374-406`), and the four-layer application scaffold is never touched (`uninstall.sh:33-37`).

## Release process

There is **no build artifact** ÔÇö a "release" is a versioned change to the framework files. The version lives in two constants in `install.sh`: `FRAMEWORK_VERSION="v1"` and `FRAMEWORK_SEMVER="1.0.0"` (`install.sh:23-24`). To cut one:
1. Branch off the integration branch (see below). Make your change to `templates/`, `install.sh`, `v1_release/`, etc.
2. Bump `FRAMEWORK_SEMVER` (and, if a file was removed/renamed, append a `DEPRECATED_SINCE` entry at `install.sh:31-33` so `--upgrade` can offer cleanup).
3. Run the suite green: `./tests/gate/run_tests.sh`.
4. Open a PR into the integration branch. Trust-root changes must get Code Owner review.
5. After merge, downstream repos pick it up by running `install.sh --upgrade` and committing the re-pinned manifest.

`UNVERIFIED:` there is no tag/GitHub-release automation in the repo ÔÇö releases appear to be branch/version-bump only. To confirm, check `git tag` and the repo's GitHub Releases page; if a formal tagging convention is desired, it is not yet codified anywhere in-tree.

## Branch strategy ÔÇö **confirm the active branch first**

`git branch --show-current` at the start of this handoff returned **`feat/token-budget-limits`** (a feature branch with in-flight work ÔÇö see `git status`). But the repo is internally inconsistent about the *integration* branch, so verify before you rely on any single source:

- `README.md:20-23` states **`develop`** is the active integration branch: "All releases are cut from `develop` ÔÇĞ Never work directly on `main`."
- The framework's own CI (`.github/workflows/framework-tests.yml:4-7`) triggers on **`init_release`, `develop`, `main`**.
- The prior handoff `handoff_cursor.md:4` names **`init_release`** as the current branch.
- `gate.sh` and `templates/ci-gate.yml` treat `main`/`master`/`develop`/`production`/`release/*` as protected in *governed* repos.

**Recommended interpretation for a new owner:** treat `develop` as the integration branch (per the front-door `README.md`), branch features off it, and never commit directly to `main`/`develop`. Reconcile the `init_release` references with the team on day one ÔÇö this is a documented inconsistency, flagged in [`08-KNOWN-ISSUES-AND-DEBT.md`](08-KNOWN-ISSUES-AND-DEBT.md). The framework enforces Gitflow-style prefixes (`feature/`, `bugfix/`, `hotfix/`, `release/`) in *governed* repos (`gate.sh:367-371`).

## Verify a repo is governed

Quick checklist for any target repo:
```bash
git config core.hooksPath                    # should be .githooks
ls .githooks/gate.sh .claude/gate_state.json .claude/gate_integrity.sha256
bash .githooks/verify_governance_integrity.sh   # should print "integrity-verified"
git config --get-all remote.origin.push      # should include refs/notes/bypasses
ls .github/workflows/gate.yml .github/CODEOWNERS
```
And confirm the human-only step: `.github/CODEOWNERS` names a real team (not `@your-org/platform-team`) **and** the default branch has "Require review from Code Owners" enabled ÔÇö without that, everything is local-only (`install.sh:1201-1213`). See [`13-OWNERSHIP-AND-CONTACTS.md`](13-OWNERSHIP-AND-CONTACTS.md).
