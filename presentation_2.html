01 — The problem

Normal prompting doesn't fail loudly.

It fails three months later, in production, when nobody remembers the chat session that introduced the debt. Here is how it actually goes wrong on a real codebase.

Failure 01 · drift

Status becomes a free-text string

The agent writes order.status = "CANCELLED" directly. No legal-transition map. An order can jump from PENDING straight to DELIVERED. The impossible state surfaces when a driver is stuck mid-delivery — not in a test.

Failure 02 · layers

Payment SDK leaks into the view

Stripe calls land in a DRF view because it's the fastest path. By session 5, ordering imports delivery imports payment. The circular dependency is now permanent in the git history.

Failure 03 · "later"

Tests get added "later"

The dispatch logic ships untested. Later never comes. The empty-driver edge case throws an unhandled 500 in production for days before anyone notices.

None of these are hypothetical. They are the exact mistakes an agent made while building our demo food-delivery app — and every one was caught before it landed.
02 — Why existing options don't close the gap

Everyone has a partial answer. None of them actually bind the agent.

A rules file / system prompt
CLAUDE.md, cursor rules, a long system prompt. Good intentions — but the model can forget them in a long session or just decide they don't apply this time.
Advisory · not enforced
Checks only in CI
Lint and tests run after the push, once the PR is open. The bad architecture is already committed, feedback is minutes-to-hours late, and merge pressure pushes it through.
Too late · after history
An LLM "reviewer" agent
A second model grades the first. It's probabilistic, burns tokens on every run, and still has no hard boundary — a confident wrong answer passes.
Probabilistic · costs tokens
What we
do instead

Deterministic checks at the commit boundary, on your machine, in under two seconds with zero LLM calls. They're scoped to the agent by the OS process tree — so a human passes instantly while the agent can't skip them. CI re-runs the identical script as the backstop, and every override is written to an append-only audit trail. Mechanical, not a vibe check.

03 — How it works

Three checks before every commit.
The agent can't skip them. You don't feel them.

01

It knows the agent from you

Walks the OS process tree. If claude-code is an ancestor, strict checks turn on. Your own shell isn't, so it passes straight through.

You → instant pass
Agent → checks on
↓
02

Bad code can't commit

ruff · mypy (strict) · import-linter boundaries · secrets scan — on the changed files. Any failure exits non-zero and the commit stops.

Clean → pass
Broken → blocked
↓
03

Big changes need a plan

Touch 5+ files and it needs a short checkpoint note first. Coverage can only go up. Functions over complexity 10 are rejected.

Note + green → commit
Missing → blocked

05 — Same feature, two ways

The status change. Prompted vs governed.

What prompting tends to write
# ordering/presentation/views.py
def update_status(request, order_id):
    order.status = request.data["status"]  ✗ any string
    if order.status == "CANCELLED":
        stripe.Refund.create(order.intent)  ✗ SDK in view
    order.save()                         ✗ no rules, no events
What the framework keeps
# ordering/domain/entities.py
def _transition(self, to: OrderStatus, now):
    if to not in LEGAL_TRANSITIONS[self.status]:
        raise ValueError(...)          ✓ illegal jump blocked
    self.status = to                     ✓ one authority
def cancel(self, actor, reason, now):
    self._transition(OrderStatus.CANCELLED, now)
    self._record(ev.order_cancelled(...)) ✓ refund via port
