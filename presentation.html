01 — The Structural Problem

The Prompt Compliance Paradox

Standard frameworks try to manage AI behavior by packing rules into a system prompt. At enterprise scale that breaks down — a model promise is not a security boundary.

Fragile Prompt Restrictions

The Illusion of Compliance

Asking a model to "always follow the layering rules" or "never skip tests" is advisory, not regulatory. Under pressure or long histories, execution becomes unpredictable — and "usually complies" is exactly what a security review will not sign off on.

Context Amnesia

On extended feature branches, large source files and tool output scroll the original rules out of the active context window. The autonomous agent drifts silently and resumes inventing patterns — with no one watching the commit where compliance quietly stopped.

History Contamination

An unconstrained autonomous agent making a blind multi-file change commits it straight into the tree. Broken baseline assumptions bleed into the permanent git log and become what the next session learns from.

ai-dev-workflow System Invariants

Inviolable OS Enforcement

Rules do not depend on the autonomous agent's willingness to listen. Enforcement lives in git hooks, process-tree checks, and shell lifecycles — where the agent has no authority to negotiate, only to comply or be blocked.

Hash-Anchored Disk Constitution

The repository rules live in CLAUDE.md, hash-anchored on disk. When that hash drifts, the gate flags it and requires the agent to re-read the blueprint before proceeding — so architectural context cannot silently decay.

Atomic Commit Barriers

A local gate intercepts the commit before code reaches the ledger. If the autonomous agent attempts a deep structural change footprint with no design brief on disk, the commit is dropped with a hard exit.

02 — System Architecture

The Enforcement Engine

Instead of steering model behavior with text, we wrap the local execution boundary in a mechanical runtime cage. Three layers, none of which ask the agent to cooperate.

01

Recursive Process-Tree Lineage Tracing

The gate never trusts an environment variable to say who is committing. On invocation it recursively climbs the OS process tree to PID 1, inspecting each ancestor's command. If the @anthropic-ai/claude-code binary appears anywhere in the lineage, the strict agent gates engage. The human engineer's shell has no such ancestor, so it is waved through with zero friction — and nested subshells cannot mask the autonomous agent, because the walk climbs unconditionally to the root.

# A wrapper/multiplexer between claude-code and git only adds parent frames;
# the loop re-targets each parent in turn, so it still reaches the real ancestor.
CURRENT_PID=$PPID
while [ "${CURRENT_PID:-0}" -gt 1 ]; do
    _AP_CMD=$(ps -p "$CURRENT_PID" -o command=)
    if echo "$_AP_CMD" | grep -qE '@anthropic-ai/claude-code'; then
        IS_AGENT=true; break                     # agent caught → strict gates engage
    fi
    _AP_PPID=$(ps -p "$CURRENT_PID" -o ppid= | tr -d ' \n')
    [ -z "$_AP_PPID" ] || [ "$_AP_PPID" = "0" ] && break   # bounded: terminates at the root
    CURRENT_PID=$_AP_PPID                            # recurse to the parent frame
done
02

Active Kill-and-Restart Graph Lifecycle

The autonomous agent's blast-radius reasoning depends on a current dependency graph; a stale map produces confident, wrong answers. This is not a passive "skip if busy" — it is active termination. During rapid atomic commit flurries, each new commit reads the tracked indexer PID from .claude/graph.pid, force-kills (kill -9) the previous build still mapping an obsolete HEAD, clears the stale PID record, and re-launches a fresh background build against the newest commit. Because exactly one indexer is ever live, two writers never collide on the same SQLite store — eliminating the SQLITE_BUSY errors that plague naive "rebuild on save" setups, while the map always tracks the live HEAD.

04

Session Integrity and Forced Handoff

Code quality degrades silently as a session ages — context falls off the attention window and the agent starts reproducing errors it already fixed. Five degradation signals (SD1–SD5) monitor continuously: redundant file reads, repeated past errors, unprompted re-narration of prior steps, uncertainty on previously resolved facts, and session-depth thresholds. When two signals fire simultaneously, the agent is required to write a full state snapshot to .claude/checkpoints/LATEST.md — capturing current phase, git SHA, files touched, decisions locked, and a cold-start RESUME INSTRUCTION — then issue the exact forced handoff message and halt. No further code is written until the human runs /clear. The next session reads LATEST.md, verifies the SHA, and resumes from the RESUME INSTRUCTION with zero context loss.

03 — De-risking and Governance

Pre-emptive Leadership Q&A

Resolving velocity, security, and maintenance concerns with the actual mechanism — not a reassurance.

QUESTION 01 "If a human clears the gate in two seconds, what stops the agent from just identifying as a human?"

The pass-through is not a self-declared identity — and the server-side CI is the authority regardless. The local gate never reads a flag that says "I am human." It inspects the real OS process ancestry; an autonomous agent's commit genuinely carries the claude-code binary as an ancestor, and the agent cannot rewrite its own parent processes from inside its sandbox. "Pretending to be human" is not a move it can make.

And local enforcement does not have to be airtight, because CI is the enforcement of record. Every push and pull request re-runs the identical gate.sh in a clean server environment — in GATE_TRIGGER=ci mode with tests forced on — no matter who or what produced the commit. Local speed is a convenience for the human engineer (a clean commit clears in under two seconds, zero network calls); the server is what actually authorizes a merge.

That two-second path is also hardened against the failure mode that breaks most git-hook rollouts: the editor deadlock. Before the bypass ever opens an interactive prompt, the hook detects non-interactive IDE and extension shells — testing [ -t 0 ], /dev/tty readability, and editor markers like TERM_PROGRAM / CURSOR_TRACE_ID — and falls through gracefully instead of hanging the terminal pane. A real terminal gets the prompt; an IDE subshell never freezes.

Process lineage: un-spoofable Final authority: server-side CI IDE shells: no terminal deadlock Human overhead: < 2s
QUESTION 02 "Can a developer just strip the hooks or erase the audit history?"

No. Local hooks are the fast path; server-side CI is the authority. If a developer strips .githooks/ or pushes with --no-verify, the committed workflow .github/workflows/gate.yml (deployed from templates/ci-gate.yml) still runs: it first verifies the governance files are present and unmodified, then re-executes the same gate.sh in GATE_TRIGGER=ci mode with tests forced on, in a clean environment the developer cannot edit. Any missing file or failed check fails the pipeline.

The emergency bypass is the opposite of Shadow IT — it is the most visible action in the repository. SKIP_GATE=1 requires a typed justification, written to a non-repudiable git note on refs/notes/bypasses, attributed to the author and replicated to the server. It is not a silent escape: the whole platform team sees every bypass via git log --show-notes=bypasses and it surfaces in CI. The pre-push hook rejects force pushes and explicitly blocks any refspec targeting refs/notes/bypasses, so git push origin :refs/notes/bypasses cannot erase the trail. A hard 24-hour resolution clock then blocks the next push until the underlying issue is fixed — a bypass is a time-boxed, audited liability, never a permanent override.

CI backstop: gate.yml Every bypass: attributed + visible Unresolved > 24h: push blocked
QUESTION 03 "AI tooling moves fast — how do we prevent configuration and documentation drift?"

One version-stamped canonical source of truth on disk. The framework ships two deployment baskets — brownfield (freezes existing debt in a baseline that can only shrink) and greenfield (zero-tolerance from commit one) — that consolidate into a single engineering constitution: CLAUDE.md.

Each repo carries a committed .mcp.json, so the moment an engineer clones, their environment inherits the identical graph configuration with zero manual setup. The graph package is pinned to an exact version, and the constitution is hash-anchored — if the rules shift, the gate flags the drift on the next run. Upgrades re-stamp the framework version while preserving every team-owned field.

Canonical rules: CLAUDE.md Zero-config onboarding: committed .mcp.json Supply chain: hash-pinned
QUESTION 04 "What if the model writes better code than the governance gate allows?"

The gate does not judge the creativity of the code — it enforces the safety and architectural boundaries of the system. Code that is technically superior but architecturally non-compliant is still a long-term maintenance risk, and the gates are precisely what stop that "innovation debt" from entering the codebase.

Better ideas are welcome — they simply travel the same human-reviewed path used to change any boundary (updating CLAUDE.md or the core_files policy via PR), rather than being smuggled in by an agent that decided the rule did not apply this time.

Gate scope: safety + architecture Prevents: innovation debt Boundary change: human PR only
04 — Core Takeaway

The Bottom Line: Scalable Isolation

By shifting AI governance out of the chat prompt and into the system environment, the repository becomes an intolerant static-analysis engine — strict on the agent, invisible to the engineer.

Zero pollution
